## 🎯 项目定位

Opc_Kit 是一个专为 AI Agent 生态设计的跨平台技能工具集。每个技能都是经过精心设计、严格验证的专业工作流，帮助产品经理、开发者、设计师更高效地完成复杂任务。

**核心理念**：
- 📐 **结构化保证** — 强制模板和验证机制确保输出质量
- ⚡ **效率优化** — 自动推断 + 按需交互减少重复工作
- 🔗 **可追溯性** — 双向追溯机制确保每个决策都有源头
- 🎓 **专业视角** — 资深专家思维框架和行业最佳实践
- 🌐 **跨平台兼容** — 平台无关设计，自动适配主流 Agent 工具链

## ✨ 功能特性

### 🎯 智能意图识别
自动识别用户意图（create/update/validate），无需手动指定工作模式。

### 🔄 双模式工作流
- **Coaching 模式（默认）** — 自主推断 → 生成完整草稿 → 自审发现缺口 → 按需交互 → 反向问题补充（~5-10 分钟）
- **Fast 模式** — 快速生成完整文档 → 反向问题补充缺失项（~2-3 分钟）

### 🔍 严格验证机制
- **第一性原理验证** — 5+ 个基础问题确保文档根基正确
- **逻辑完整性审查** — US↔FR 双向追溯、追踪↔指标追溯
- **边界与风险扫描** — 主动识别异常流程、边界条件、外部依赖

### 📝 12 章标准模板 + 3 章自动生成
- 固定骨架：问题描述、目标定义、目标用户、用户故事、功能交互流程图、详细功能列表、功能详情、追踪设计、未来改进计划、风险与依赖
- 自动生成：决策日志、术语表、假设索引

### 🌐 自动语言适配
智能检测用户对话语言（中英文），自动生成对应语言的 PRD 文档。用户显式指令优先（"用英文写" / "write this in Chinese"）。

### 📊 质量评分体系
7 维度量化评分（满分 100），含生产就绪基准线（70+ 分 = 生产就绪，85+ 分 = 优秀）。

### 🎯 进度追踪
自动生成任务清单，实时播报进度（"✅ 第 1-3 章已完成。正在生成用户故事..."）。

### 💡 推荐驱动交互
每次交互提供 1-3 个经过深思熟虑的推荐选项及理由，引导用户做决策而非填空。

## 📚 create-prd 技能详解

### 核心功能

**create-prd** 是一个专业的 PRD（产品需求文档）写作助手，提供从创建、更新到验证的完整工作流支持。

| 意图       | 功能            | 触发信号                                    |
| ---------- | --------------- | ------------------------------------------- |
| **create**   | 创建新 PRD      | "新建 PRD"、"撰写需求文档"、"创建产品需求"、"Create a PRD"  |
| **update**   | 更新已有 PRD    | "更新/修改已有 PRD"、"PRD 变更"、"在现有文档上增加功能"、"Update docs/specs/xxx.md" |
| **validate** | 验证 PRD 完整性 | "验证/检查 PRD"、"审查需求文档完整性"、"Validate docs/specs/xxx.md"        |

### 工作模式（create/update）

| 模式     | 行为                                                                                                 |
| -------- | ---------------------------------------------------------------------------------------------------- |
| **coaching** | 先根据上下文自主推断所有章节内容生成初稿 → 自评审发现断裂/缺失 → 条件化交互 → 反向追问补全非关键推断 |
| **fast**     | 基于已有信息直接生成 PRD → 反向追问缺失项                                                            |

#### Coaching 交互规则
- 每次向用户提问时，必须附 1-3 条经过深思熟虑的推荐选项（不超过 3 条）
- 每条推荐选项需标注推荐理由，引导用户做选择题而非填空题
- 用户回答后必须确认再推进

#### 轻量实时确认（coaching 模式）

| 推断类型   | 处理方式                          | 示例                               |
| ---------- | --------------------------------- | ---------------------------------- |
| **关键假设**   | 当场轻声确认，不标记 [ASSUMPTION] | "我假设目标用户是经纪人，对吗？"   |
| **非关键推断** | 标记 `[ASSUMPTION: xxx]`，不打断    | PRD 中自动标记，最终汇总到反向追问 |

**关键假设定义（8 类）**：用户角色、成功指标量化值、技术约束、权限范围、数据量级、业务规则（状态机/限制/审批链）、合规与安全要求（审计日志/数据驻地/GDPR）、当前替代方案/用户行为基线。

#### 模式切换指令

| 用户指令示例                                        | 命中模式 |
| --------------------------------------------------- | -------- |
| "用 fast/fast path/快速/直接生成/别问了直接写/skip" | fast     |
| "用 coaching/coach/引导/一次一问/慢慢来"            | coaching |

### 16 条核心原则

1. **模板固定** — 12 章骨架（CN）/ 12 章骨架（EN） + 自动生成章节，章节顺序不可调整
2. **强校验不可跳过** — US↔FR 双向追溯、第6↔第7章 1:1 对应、埋点↔成功指标追溯、指标→计算方式 1:1 追溯
3. **变更记录强制** — update 意图必须先追加变更记录条目，否则禁止保存
4. **流程图必有** — 至少 1 张 mermaid 流程图，核心场景独立图；**API 调用/数据查询/外部依赖必须有失败+超时分支，不可只有成功路径**
5. **验收标准可测试** — 每条必须含可量化/可执行判定条件
6. **[ASSUMPTION] 标签强制** — 推断内容必须标记，完成后自动汇总到假设索引
7. **一次一问**（coaching 模式交互时）：不批量追问
8. **异常路径覆盖** — 流程图中每个 API 调用/外部依赖节点必须有成功/失败/超时三条分支，用户操作节点必须有异常路径（网络断开、权限不足、数据不存在）
9. **深度推理前置** — 生成 PRD 初稿前，必须先进行深度推理分析（用户动机、商业价值、技术可行性、风险矩阵），将推理结果融入 PRD 内容而非直接询问用户
10. **高级 PM 视角** — 以经验丰富的高级产品经理标准引导和思考，在问题描述、目标定义章节引入产品思维框架（Why Now、差异化、用户分层、商业价值），在风险与依赖章节提供行业最佳实践建议
11. **推荐选项驱动** — 每次与用户互动时，给出经过深思熟虑的推荐建议（原则上不超过 3 条），每条有理有据，不得胡编，明确标注推荐项，让用户做决策
12. **语言自适应** — 检测用户语言，匹配 PRD 输出语言；用户明确指令优先级最高
13. **默认路径约定** — 自动保存到 `docs/specs/`，命名清晰；保存前确认路径
14. **强制进度追踪** — 开始时创建 TodoWrite 任务清单，每章完成后更新状态
15. **跨 Agent 兼容** — 平台无关技能，支持 OpenCode、Claude Code、Cursor、Codex。工具调用使用通用描述，各 Agent 根据自身工具集映射执行
16. **环境检测优先** — 运行校验脚本前自动检测 Node.js 环境

### 需求评审流程

无论 fast 或 coaching 模式，PRD 初稿输出后**必须**执行三步评审流程。

#### 第一步：第一性原理验证（5+ 追问）

1. **用户是谁？** —— 目标用户是否明确、可描述？
2. **他要什么？** —— 用户故事是否覆盖了真实场景？
3. **用户为什么现在要？** —— 问题紧急性是否成立（Why Now）？
4. **用户为什么用你的方案？** —— 差异化价值是否清晰？
5. **你怎么知道自己做对了？** —— 成功指标是否可量化、可执行？
6-10+ （根据产品复杂度补充，至少 5 个）

#### 第二步：逻辑完整度审查

| 追溯方向       | 规则                                       | 要求    |
| -------------- | ------------------------------------------ | ------- |
| US→FR          | 每条用户故事至少被一个功能实现             | 100%    |
| FR→US          | 每个功能至少回应一条用户故事               | 100%    |
| 埋点→指标      | 每条埋点服务于至少一个成功指标             | 100%    |
| 指标→计算方式  | 每个成功指标在第 8.3 节有对应计算方式      | 100%    |
| 指标→问题      | 每个成功指标能回答"是否解决了痛点"         | 100%    |

**断裂点检测**：流程中任何环节无法承接到上一环节即标记断裂点。

#### 第三步：边界与风险扫描

| 类别       | 探针问题                               | 示例                                       |
| ---------- | -------------------------------------- | ------------------------------------------ |
| 异常流程   | "如果用户做 X 操作失败会怎样？"        | 收藏时网络断开、存储空间满                 |
| 边界条件   | "最多收藏多少？并发调用会怎样？"       | 收藏上限、并发调用限制、数据量级           |
| 外部依赖   | "哪个外部接口还没对接？有备选方案吗？" | 地图 API、用户权限系统、分享服务           |
| 不可控因素 | "用户不按预期使用怎么办？"             | 用户收藏 1000+ 内容、分享权限滥用          |

### 方法论参考

- **第一性原理**（First Principles）：Elon Musk / Aristotle — 剥离表象，回到不可再简化的基本事实
- **逆向工作法**（Working Backwards）：Amazon/PRFAQ — 从用户痛点出发反推
- **YAGNI 原则**（You Ain't Gonna Need It）：XP 极限编程 — 不做多余的事
- **MoSCoW 优先级**：Must/Should/Could/Won't — 确保 v1 只做必须的事

## 📊 PRD 质量评分体系

每份 PRD 在 7 个维度上获得评分（满分 100 分）：

| 维度                        | 权重 | 满分 | 说明                                      |
| --------------------------- | ---- | ---- | ----------------------------------------- |
| 完整性 Completeness         | 20%  | 20   | 12 章齐全且已填充                         |
| 追溯性 Traceability         | 20%  | 20   | US↔FR 1:1 映射，无孤立需求                |
| 可测试性 Testability        | 15%  | 15   | 验收标准可量化/可执行                     |
| 清晰性 Clarity              | 15%  | 15   | 无歧义用语，术语清晰                      |
| 异常覆盖 Exception Coverage | 10%  | 10   | 所有外部调用均有失败路径                  |
| 指标对齐 Metrics Alignment  | 10%  | 10   | 追踪事件与成功指标 1:1 映射               |
| 风险管理 Risk Management    | 10%  | 10   | 依赖、风险、缓解措施齐备                  |

**评分基准**：
- **70+ 分** → 生产就绪
- **85+ 分** → 优秀
- **< 70 分** → 需修订后再分享给利益相关者

## 🔗 工具集成

生成的 PRD 可直接与主流产品开发工具集成：

- **Figma** — PRD 元数据中附带原型 URL（`metadata.prototype`）
- **Jira** — 功能列表可导出为 Epic/Story 结构（F-x.x → Story ID 映射）

## 🌐 跨平台兼容

本技能为平台无关设计，自动适配主流 AI 编程代理：

### 工具映射

| 操作       | OpenCode         | Claude Code | Cursor | Codex      |
| ---------- | ---------------- | ----------- | ------ | ---------- |
| 文件读取   | `read`             | `Read`        | 内置   | `read_file`  |
| 文件写入   | `write`            | `Write`       | 内置   | `write_file` |
| 精确编辑   | `edit`             | `Edit`        | 内置   | `edit_file`  |
| 文件查找   | `glob`             | `Glob`        | 内置   | `glob`       |
| 内容搜索   | `grep`             | `Grep`        | 内置   | `search`     |
| 大文件处理 | `ctx_execute_file` | `Bash`        | `Bash`   | `run_shell`  |
| 任务管理   | `todowrite`        | `TodoWrite`   | 内置   | 内置       |
| 子代理分发 | `task`             | `Task`        | ❌     | ❌         |

### 降级策略

当特定工具不可用时：
1. **大文件分析** → 用 `bash`/`run_shell` + `head`/`tail`/`grep` 读取关键段落
2. **子代理** → 当前会话顺序执行，每步完成后报告进度
3. **任务清单** → 会话开始时文本列计划，逐步标记完成
