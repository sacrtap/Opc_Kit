# 埋点设计规则

## 双向追溯强校验

### 正向追溯（成功指标 → 埋点）

第 2 章每个成功指标必须满足：
1. 在第 8.3 节"成功指标计算方式"中有计算公式
2. 计算公式中引用了至少一个埋点编号（BT-x.x）

**缺失 → Critical**：指标无法测量，PRD 目标不可验证。

### 反向追溯（埋点 → 成功指标）

第 8.2 章每条埋点必须满足：
1. "对应成功指标"列非空
2. 引用的成功指标在第 2 章中存在

**为空 → Critical**：埋点无服务对象，违反 1:1 强制规则，必须删除或关联到成功指标。

## 埋点编号格式

`BT-{大版本号}.{序号}`

示例：
- `BT-2.1`：第 2 个优化项的第 1 个埋点
- `BT-3.1`：第 3 个优化项的第 1 个埋点

大版本号从 PRD 标题或 metadata 推断。

## 埋点清单表格式

| 字段 | 说明 | 级别 |
|------|------|------|
| 埋点编号 | BT-x.x 格式 | Critical |
| 事件名称 | 蛇形命名，如 `auto_fill_query` | Critical |
| 触发时机 | 具体何时触发，对应哪个用户操作 | Critical |
| 对应成功指标 | 第 2 章指标名称 | Critical |
| 关键业务字段 | 字段名 + 枚举值列表 | Critical |

### 关键业务字段格式

列出字段名和可能的枚举值：

```
parse_status（success/partial/failed/pending）、
gps_status（success/failed/timeout）、
response_time（响应时间ms）
```

## 成功指标计算方式表格式

| 字段 | 说明 |
|------|------|
| 成功指标 | 第 2 章指标名称 |
| 计算公式 | 具体计算方式，如"所有搜索会话的操作步数总和 / 搜索会话总数" |
| 使用埋点 | 埋点编号，如 BT-2.6（step_count 字段求平均） |

## CSAT 调研方案（可选）

| 字段 | 说明 |
|------|------|
| 调研时机 | 上线后第 N 天推送 |
| 调研方式 | 应用内弹窗问卷 / 邮件调研 / 电话访谈 |
| 目标样本量 | N≥{数量} |
| 调研对象 | 去重用户 / 特定行为用户 |
| 核心问题 | 具体问题文案 |

## 埋点设计原则

1. **每个指标至少一个埋点**：不可测量的指标不是指标
2. **每个埋点服务于指标**：不为埋点而埋点
3. **1:1 强制对应**：**所有点必须 1:1 服务于第 2 章至少一个成功指标**。不与成功指标直接相关的交互行为（如纯 UI 操作、Toast 展示等）不纳入埋点表
4. **业务字段完整**：埋点必须包含计算指标所需的所有字段
5. **事件命名一致**：使用蛇形命名，语义清晰
6. **触发时机明确**：对应具体用户操作或系统事件
