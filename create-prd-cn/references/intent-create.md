# create 意图详细流程

## Step 1: 意图识别 + 工作模式选择

1. 扫描用户首条消息，检测是否包含模式切换指令（fast/coaching 关键词）
2. 若包含显式指令 → 直接采用指定模式
3. 若未明确指定 → **默认 coaching 模式**
4. 用户可随时切换模式

## Step 2: Metadata 自动推测

从对话上下文自动推测填充所有 Metadata 字段，仅在完全无法推测时统一询问一次。

| 字段 | 推断来源 | 推测策略 |
|------|---------|---------|
| 作者 | 对话提及 / git config user.name / 历史会话记忆 | 优先使用对话中明确提到的作者名 |
| 状态 | 固定 | `Draft` |
| 创建日期 | 系统日期 | 今日 |
| 最后更新 | 系统日期 | 今日 |
| 版本 | 新建固定 | `v1.0.0` |
| 项目 | 需求名称 / 对话提及 | 从用户首条消息的需求名称中提取 |
| 关联文档 | 默认 | "无" |
| 原型文件 | 默认 | "无" |

**统一问规则**：仅当作者和项目字段完全无法推测时，统一询问一次：
> "请填写 Metadata 信息：作者是谁？项目名称是什么？（关联文档和原型文件如无默认填"无"）"

## Step 3a: fast 模式流程

1. 用户贴大段背景/需求描述 → 直接生成 PRD 全文
2. 推断内容用 `[ASSUMPTION: xxx]` 标记
3. 生成后逐章节展示，追问缺失项：
   - 成功指标量化值
   - 验收标准可测试条件
   - 流程图分支细节
   - 外部依赖排期状态
4. 用户确认后进入 Step 4

## Step 3b: coaching 模式流程（重构版）

**新流程：先自主推断 → 条件化交互 → 反向追问补全**

### 第一步：自主推断初稿

基于用户首条消息和所有可见上下文，以高级产品经理视角对所有章节内容进行合理推断：

1. **深度推理分析**（生成前必做）：
   - 用户动机分析：用户真正要解决的问题是什么？背后的商业价值是什么？
   - Why Now 验证：为什么是现在做这个功能？有没有时间窗口？
   - 用户分层：目标用户画像是否明确？有没有次要用户群体？
   - 技术可行性：方案在技术上是否可实施？最大风险是什么？
   - 风险矩阵：列出前 3 个可能的风险及缓解方案

2. **自主推断所有章节内容**：不要逐章询问，而是基于已有信息一次性完成 12 章的初稿内容推断

3. **自评审**：运行强校验清单，检查 US↔FR 追溯、埋点↔指标追溯、指标→计算方式追溯等

### 第二步：条件化交互

自评审完成后，根据发现的问题类型决定是否与用户交互：

| 发现问题类型 | 处理方式 |
|-------------|---------|
| **关键假设缺失**（用户角色、成功指标量化值、技术约束、权限范围、数据量级） | 必须与用户确认，附 1-3 条推荐选项 |
| **不确定性会影响后续章节准确性**（如触发机制不确定导致流程图无法设计） | 优先补完后再继续 |
| **非关键推断**（交互细节、UI 样式偏好、默认参数值、异常流程处理方式） | 标记 `[ASSUMPTION: xxx]`，不打断，汇总到反向追问 |

**推荐选项规则**（每次向用户提问时必须遵守）：
- 附 1-3 条经过深思熟虑的推荐选项（不超过 3 条）
- 每条附推荐理由，让用户理解为什么推荐这个
- 明确标注"推荐项"，引导用户做选择题而非填空题
- 示例："配置更新的触发方式，你倾向于哪种？
  - **A. 轮询方式（推荐）**：App 定期（如每 5 分钟）向服务端检查配置最新版本 → 推荐理由：实现简单，不依赖推送基础设施，适合不定期更新的场景
  - **B. 推送方式**：服务端配置更新后，通过推送消息/WebSocket 通知 App → 需要额外的推送服务支持
  - **C. 混合方式**：以轮询为主，推送为辅 → 适合高频更新且对实时性要求高的场景"

### 第三步：生成 PRD 初稿 + 反向追问

所有关键假设确认完成后，生成完整 PRD 初稿。在末尾附上"反向追问"段落：

```markdown
## 反向追问

以下内容为推断或非关键待确认项，需要你确认或补充：

1. **[ASSUMPTION]** {推断内容 1} — {追问问题}
2. **[ASSUMPTION]** {推断内容 2} — {追问问题}
...

你可以针对具体章节说"重新写第 X 章"或"补充 Y 细节"。
```

**规则：**
- 只列出未确认的 [ASSUMPTION] 标记（关键假设已在交互中确认，不列入）
- 每条追问附一个具体问题，引导用户确认
- 列出对话中推断但用户未明确确认的内容
- 输出 PRD 质量评分（按 `references/scoring-rules.md` 执行）

## Step 4: 流程图生成

1. 根据交互逻辑识别业务场景
2. 提议拆分方案（按场景/数据流向，单图 8-20 节点）
3. 逐张生成 mermaid 代码，每张要求用户确认
4. 校验：节点数、语法、状态值与数据表一致性

规则详见 `references/mermaid-rules.md`

## Step 5: 强校验

生成后自动运行以下校验：

1. US ↔ F-x.x 双向追溯（列出孤立项）
2. 第 6 章 vs 第 7 章 1:1 对应（列出缺失项）
3. 验收标准可测试性校验（标记不可测试条目）
4. 埋点 ↔ 成功指标双向追溯
5. 第 2 章成功指标数量 vs 第 8.3 节计算方式数量 1:1 对应
6. 流程图状态值与数据表字段对齐
7. 外部依赖排期状态字段合规
8. PRD 质量评分（7 维度）

## Step 6: 需求评审（强制）

PRD 初稿输出后，必须执行需求评审流程：

1. **第一性原理验证**（5+ 追问）：逐一验证用户、需求、紧急性、差异化、成功指标
2. **逻辑完整度审查**：全链路追溯（问题→目标→US→FR→流程图→埋点→指标→计算方式）
3. **边界与风险扫描**：异常流程、边界条件、外部依赖、不可控因素
4. **超过 10 个追问时**：10 轮完成后询问用户是否再次深度评审

评审结果以固定格式附在 PRD 末尾（不插入主文档）。

规则详见 `references/review-rules.md`

## Step 7: 术语表 + 假设索引自动生成

1. 扫描文档，提取跨章节出现 ≥2 次的领域名词 → 候选术语表
2. 扫描 `[ASSUMPTION]` 标签 → 汇总到第 12 章
3. 向用户展示候选术语表，确认入表项

## Step 8: 保存

1. 询问保存路径（默认 `docs/{产品}_prd/PRD-{产品}-{特性}.md`）
2. 写入文件，返回路径
3. 提示用户可继续 update 或 validate
