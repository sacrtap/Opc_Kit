# 各章节内嵌规则汇总

## user-story 规则

### 句式强制

每条用户故事必须使用以下句式：
```
作为{角色}，我希望{行为}，以便{价值}
```

- **角色**：具体的用户类型（如"经纪人"、"众趣外包编辑"），不使用"用户"这种泛称
- **行为**：可执行的操作（如"打开户型图工具时系统自动填好当前房源的省市和小区名"）
- **价值**：用户获得的好处（如"只需确认或微调即可开始搜索"）

### 验收标准

每条 US 必须有对应验收标准列，内容可量化或可执行验证。

### 双向追溯

**US → F-x.x**：每条 US 至少被第 7 章一个 F-x.x 实现。
- 验证方法：扫描 F-x.x 的功能描述和交互说明，查找是否提及该 US 的场景
- 孤立 US 报告为 Critical

**F-x.x → US**：每个 F-x.x 至少回应一条 US。
- 验证方法：检查 F-x.x 是否服务于某个 US 描述的用户场景
- 无来源 F-x.x 报告为 Warning（可能是合理的扩展功能，但需说明理由）

## feature-detail 规则

### 每个 F-x.x 必含字段

| 字段 | 说明 | 级别 |
|------|------|------|
| 功能描述 | 一句话描述功能做什么 | Critical |
| 触发时机 | 何时触发/在什么场景下执行 | Critical |
| 交互说明 | 具体交互步骤/状态变化/数据流转 | Critical |
| 验收标准 | `- [ ]` 格式 + 可测试条件 | Critical |
| 场景行为表 | 多状态场景时必须有（可选） | Warning |

### 验收标准可测试性

每条验收标准必须包含以下至少一种可测试条件：

| 类型 | 示例 |
|------|------|
| 量化阈值 | `P95 <1s`、`>10s 视为超时`、`>90% 成功率` |
| 可执行验证 | `INSERT 语句执行成功`、`Redis GET 返回结果` |
| 状态值 | `parse_status = success`、`gps_status 标记为 failed/timeout` |
| UI 观测 | `提示条单行显示，不折行`、`清除按钮功能正常` |

**禁止**：模糊描述如"功能正常"、"用户体验好"、"运行稳定"。

### 场景行为表

适用于有多个状态分支的功能（如 parse_status 有 success/partial/failed/pending 四种场景）：

| 场景 | 行为 |
|------|------|
| {条件 A} | {具体行为描述} |
| {条件 B} | {具体行为描述} |

场景值必须与数据表 ENUM 定义一致。

## risk-dependency 规则

### 技术风险（10.1）

| 字段 | 规则 |
|------|------|
| 风险编号 | R-{序号} 格式 |
| 影响程度 | 固定枚举：高/中/低 |
| 缓解措施 | 必须有具体方案，不能为空 |

### 外部依赖（10.2）

| 字段 | 规则 |
|------|------|
| 依赖编号 | D-{序号} 格式 |
| 排期状态 | **强制字段**，值必须在以下枚举内：待对接/待评审/待确认/已确认/已完成/已阻塞 |

缺失排期状态或值不在枚举内时，validate 报告为 Critical。

### 已知限制（10.3）

| 字段 | 规则 |
|------|------|
| 限制编号 | L-{序号} 格式 |
| 解决计划 | 必须有，不能为空。可填"已列入未来改进计划" |

## 问题表规则（第 1 章）

### 具体问题表

| 字段 | 规则 |
|------|------|
| # | 序号，从 1 开始 |
| 问题 | 具体问题描述 |
| 用户反馈 | 用户原话或典型反馈，用引号包裹 |
| 严重程度 | P0/P1/P2，P0 不超过 3 条 |

P0 问题必须对应第 2 章的核心目标。

## 目标定义规则（第 2 章）

### 成功指标表

| 字段 | 规则 |
|------|------|
| 指标 | 指标名称，可量化 |
| 当前基线 | 当前值或 "N/A" |
| 目标 | **必须量化**（不允许"提升用户体验"） |
| 测量方式 | 埋点编号或调研方式 |

### counter-metrics 建议

当存在可能被优化的主指标时，建议添加反向指标（counter-metric）：
- 例：主指标"自动填充采纳率 >80%" → 反向指标"用户手动修改率不应上升"
- 防止优化单一指标而损害其他体验

## 未来改进计划规则（第 9 章）

| 字段 | 规则 |
|------|------|
| 编号 | **必须延续主功能编号**，如本期最后 F-2.12，下期从 F-2.13 开始 |
| 改进项 | 具体描述 |
| 原因 | 为什么本期不做 |
| 优先级 | P0/P1/P2 |
| 预计迭代 | 目标版本号（如 v2.0） |

不允许从无关编号开始（如本期 F-2.12，下期从 F-3.1 开始），除非用户明确说明跨大版本。

## 功能优先级定义

所有功能优先级必须使用 P0/P1/P2 三级体系：

| 级别 | 定义 | 含义 |
| ---- | ---- | ---- |
| **P0** | 核心/阻塞 | 不做则功能不可用，v1 必须交付 |
| **P1** | 重要 | 用户体验显著受影响，v1 应尽量交付 |
| **P2** | 锦上添花 | 有更好，没有也不影响核心链路 |

**规则：**
- 不使用 Must/Should/Could/Won't 作为优先级标签
- P0 功能应限制数量（建议每次发布 ≤5 个）
- P2 功能如果时间紧张可以延期到后续迭代
