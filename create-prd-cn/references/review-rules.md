# 需求评审规则

## 触发时机

无论 fast 或 coaching 模式，PRD 初稿输出后**必须**执行需求评审流程。

## 评审流程

### 第一步：第一性原理验证（5+ 追问）

回到本质，逐一追问以下核心问题：

| # | 追问                     | 验证标准                                     | 失败处理                     |
| - | ------------------------ | -------------------------------------------- | ---------------------------- |
| 1 | **用户是谁？**           | 目标用户是否明确、可描述、有具体画像         | 标记 [ASSUMPTION] 要求确认   |
| 2 | **他要什么？**           | 用户故事是否覆盖了真实场景，不是臆想需求     | 列出缺失的用户故事           |
| 3 | **用户为什么现在要？**   | 问题紧急性是否成立（Why Now），是否有时间窗口 | 标记 [ASSUMPTION] 要求确认   |
| 4 | **用户为什么用你的方案？** | 差异化价值是否清晰，与现有方案的区别         | 列出竞品对比或差异化说明     |
| 5 | **你怎么知道自己做对了？** | 成功指标是否可量化、可执行、有基线和目标值   | 补充量化指标或标记 [ASSUMPTION] |

**扩展追问（根据产品复杂度补充，至少 5 个）：**

6. **这个功能的杠杆效应是什么？** —— 是否一个小改动带来大收益（LNO 框架）
7. **如果砍掉 50% 功能，保留什么？** —— 验证 MoSCoW 优先级是否合理
8. **用户不按预期使用怎么办？** —— 是否有降级方案或备选路径
9. **技术实现的最大风险是什么？** —— 是否有技术可行性验证
10. **3 个月后项目失败，原因是什么？** —— Pre-mortem 事前验尸

### 第二步：逻辑完整度审查

按以下路径走一遍全链路追溯：

```
问题描述 → 目标定义 → 用户故事 → 功能清单 → 流程图 → 埋点 → 成功指标
```

**追溯规则：**

| 追溯方向     | 规则                               | 通过率要求 |
| ------------ | ---------------------------------- | ---------- |
| US→FR        | 每条用户故事至少被一个功能实现     | 100%       |
| FR→US        | 每个功能至少回应一条用户故事       | 100%       |
| 埋点→指标    | 每条埋点服务于至少一个成功指标     | 100%       |
| 指标→计算方式 | 每个成功指标在 8.3 节有对应计算方式 | 100%       |
| 指标→问题    | 每个成功指标能回答"是否解决痛点"   | 100%       |

**断裂点检测：**
- 检查每个环节是否承接上一环节
- 是否有功能没有对应的用户故事（过度设计）
- 是否有用户故事没有对应的功能（需求遗漏）
- 是否有埋点没有对应的成功指标（无效埋点）
- 是否有成功指标没有对应的埋点（无法验证）

### 第三步：边界与风险扫描

主动询问用户以下 4 类边界条件：

| 类别       | 探针问题                               | 示例                                   |
| ---------- | -------------------------------------- | -------------------------------------- |
| 异常流程   | "如果用户做 X 操作失败会怎样？"        | 收藏时网络断开、存储空间已满           |
| 边界条件   | "最多收藏多少？并发调用会怎样？"       | 收藏上限、并发调用限制、数据量级       |
| 外部依赖   | "哪个外部接口还没对接？有备选方案吗？" | 地图 API、用户权限系统、分享服务       |
| 不可控因素 | "用户不按预期使用怎么办？"             | 用户收藏 1000+ 户型、分享权限滥用      |

### 超过 10 个追问时的处理

当追问数量超过 10 个时，10 个问题处理完成后询问用户：

> "已完成 10 轮追问评审。是否需要再次进行深度评审？(yes/no)"

用户选择 yes → 继续下一轮追问（最多 20 个）
用户选择 no → 结束评审，输出评审记录

## 输出格式

评审结果以固定格式附在 PRD 末尾（不插入主文档）：

```markdown
## 评审记录

### 第一性原理验证
- 用户是谁：✅ 明确 / ❓ 待确认（附 [ASSUMPTION] 列表）
- 他要什么：✅ 覆盖 / ❓ 缺失（列出缺失的用户故事）
- 为什么现在要：✅ 成立 / ❓ 存疑
- 为什么用你的方案：✅ 清晰 / ❓ 模糊
- 怎么知道做对了：✅ 可量化 / ❓ 主观
- （扩展追问 6-10+ 按需补充）

### 逻辑完整度
- 断裂点：（无 / 列出断裂的具体环节）
- US→FR 追溯通过率：X/Y（XX%）
- FR→US 追溯通过率：X/Y（XX%）
- 埋点→指标追溯通过率：X/Y（XX%）
- 指标→计算方式追溯通过率：X/Y（XX%）
- 指标→问题追溯通过率：X/Y（XX%）

### 边界与风险
- 异常流程：（列出待补充场景）
- 边界条件：（列出待确认参数）
- 外部依赖：（列出未确认依赖）
- 不可控因素：（列出待讨论场景）
```

## 方法论参考

| 方法论             | 来源              | 核心思想                                   | 应用场景                     |
| ------------------ | ----------------- | ------------------------------------------ | ---------------------------- |
| 第一性原理         | Elon Musk / Aristotle | 剥离表象，回到不可再简化的基本事实         | 验证需求本质是否成立         |
| Working Backwards  | Amazon/PRFAQ      | 从用户痛点出发反推解决方案                 | 确保需求从用户视角出发       |
| YAGNI 原则         | XP 极限编程       | You Ain't Gonna Need It，不做多余的事      | 砍掉过度设计的功能           |
| MoSCoW 优先级      | DSDM              | Must/Should/Could/Won't 分级               | 确保 v1 只做必须的事         |
| Pre-mortem 事前验尸 | Shreyas Doshi     | 假设 3 个月后项目失败，倒推原因            | 提前识别致命风险             |
| LNO 框架           | Shreyas Doshi     | Leverage/Neutral/Overhead 评估功能杠杆效应 | 判断功能是否有足够杠杆效应   |
| 双钻模型           | Design Council    | 先发散再收敛，确保探索多种方案             | 验证是否探索过替代方案       |

## 追问库（按需调用）

### 用户相关追问
- 目标用户的使用场景是什么？在什么环境下使用？
- 用户的技术水平如何？需要降低使用门槛吗？
- 用户群体有多大？是内部工具还是面向所有客户？
- 用户现有的替代方案是什么？为什么不用？

### 价值相关追问
- 这个功能带来的商业价值是什么？
- 如果不做这个功能，会失去什么？
- 这个功能的杠杆效应是什么？（一个小改动带来大收益）
- 如果砍掉 50% 功能，保留什么？

### 技术相关追问
- 技术实现的最大风险是什么？
- 有技术可行性验证吗？
- 外部依赖的排期状态是什么？
- 有备选方案吗？

### 风险相关追问
- 用户不按预期使用怎么办？
- 数据量级增长后会怎样？
- 并发场景下会怎样？
- 3 个月后项目失败，原因是什么？
