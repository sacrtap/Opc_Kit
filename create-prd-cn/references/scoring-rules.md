# PRD 质量评分体系

## 评分规则

PRD 生成并执行强校验后，必须输出 7 维度质量评分，满分 100 分。

### 维度定义

| 维度                | 权重 | 满分 | 评估内容                                       | 评分方法                                                                 |
| ------------------- | ---- | ---- | ---------------------------------------------- | ------------------------------------------------------------------------ |
| **A. 内容完整度**       | 30%  | 30   | 10 章覆盖率、Metadata 字段完整性、变更记录完整性 | 每缺失 1 章扣 5 分；Metadata 每缺失 1 字段扣 2 分；无变更记录扣 3 分 |
| **B. 追溯通过**          | 25%  | 25   | US↔FR 双向追溯、埋点↔指标↔计算方式三向追溯       | 每项追溯通过率 × 对应权重占比（US→FR 8分，FR→US 8分，埋点↔指标 5分，指标→计算方式 4分） |
| **C. 验收标准可测试性**    | 15%  | 15   | 每条验收标准是否含可量化/可执行条件，无模糊词      | 可测试验收标准数 / 总验收标准数 × 15                                     |
| **D. 流程图合规性**        | 15%  | 15   | 是否存在 mermaid、语法是否正确、是否有异常/失败分支 | 有图且语法正确得 5 分；每个 API/依赖节点有失败分支得 5 分；每个判断节点有降级策略得 5 分 |
| **E. [ASSUMPTION] 覆盖率** | 10%  | 10   | 所有推断是否标记并汇总到假设索引                | 已标记数 / 实际推断数（≥2 次）× 5；索引表已汇总得 5 分                      |
| **F. 评审记录完整性**      | 10%  | 10   | 是否执行三评审步骤并输出固定格式（第一性原理+逻辑完整度+边界风险） | 每完成一步得 3 分，固定格式输出得 1 分                                    |
| **G. 产品思维深度**     | 10%  | 10   | Why Now 是否清晰、差异化价值是否明确、用户分层是否合理、商业价值是否阐述、风险预判是否全面 | Why Now 明确得 2 分；差异化清晰得 2 分；用户分层明确得 2 分；商业价值阐述得 2 分；风险预判全面得 2 分 |

**综合得分 = Σ（维度得分 × 权重）**

### 等级划分

| 综合得分    | 等级   | 说明                                     |
| ----------- | ------ | ---------------------------------------- |
| **90-100**  | 优秀   | 内容完整、追溯无断裂、评审流程执行到位，可直接进入评审 |
| **75-89**   | 良好   | 基本完整，有少量非关键项待补充，建议快速修正 |
| **60-74**   | 及格   | 关键链路完整，但有多项待确认/补充，需迭代完善 |
| **<60**     | 待定   | 缺失较多关键内容，建议重新梳理后再生成     |

### 评分输出格式

在 PRD 末尾或单独输出评分结果：

```markdown
## PRD 质量评分

| 维度                | 权重 | 得分 | 扣分说明                               |
| ------------------- | ---- | ---- | -------------------------------------- |
| A. 内容完整度       | 30%  | X/30 | {具体扣分原因}                         |
| B. 追溯通过         | 25%  | X/25 | {具体追溯断裂点}                       |
| C. 验收标准可测试性 | 15%  | X/15 | {不可测试条目数}                       |
| D. 流程图合规性     | 15%  | X/15 | {缺失分支/节点数}                      |
| E. [ASSUMPTION] 覆盖率 | 10% | X/10 | {未标记推断数/未汇数}               |
| F. 评审记录完整性   | 10%  | X/10 | {未执行的评审步骤}                     |
| G. 产品思维深度     | 10%  | X/10 | {Why Now/差异化/用户分层/商业价值/风险} |
|                     |      |      |                                        |
| **综合得分**        | 100% | **XX/100** | **等级：优秀/良好/及格/待定**      |
```

### 各维详细评分标准

#### A. 内容完整度（30 分）

| 检查项                       | 分值 | 扣分规则                          |
| ---------------------------- | ---- | --------------------------------- |
| 第 1-10 章全部存在           | 20   | 每缺失 1 章扣 5 分                 |
| Metadata 8 字段完整          | 5    | 每缺失 1 字段扣 2 分，扣完为止      |
| 变更记录存在且有内容         | 5    | 无变更记录扣 3 分；空记录扣 1 分   |

#### B. 追溯通过（25 分）

| 追溯项           | 分值 | 计算方式                                       |
| ---------------- | ---- | ---------------------------------------------- |
| US→FR 追溯通过率 | 8    | 通过率 × 8                                     |
| FR→US 追溯通过率 | 8    | 通过率 × 5                                     |
| 埋点↔指标追溯      | 5  | 双向通过率均值 × 5                             |
| 指标→计算方式追溯  | 4    | 通过率 × 4                                     |

#### C. 验收标准可测试性（15 分）

| 检查项                         | 分值 | 评分方法                                   |
| ------------------------------ | ---- | ------------------------------------------ |
| 模糊词黑名单检测               | -    | 命中黑名单词汇的每条验收标准标记为不可测试 |
| 可测试判定条件                 | 15   | 可测试数 / 总验收标准数 × 15               |

**模糊词黑名单**：`功能正常`、`用户体验好`、`运行稳定`、`友好`、`正常显示`、`操作流畅`、`用户满意`、`美观`、`直观`

#### D. 流程图合规性（15 分）

| 检查项                              | 分值 | 评分方法                               |
| ----------------------------------- | ---- | ------------------------------------------ |
| 至少 1 张 mermaid 图，语法合规      | 5    | 有图且语法正确得 5 分，否则 0 分           |
| 每个 API/依赖节有失败/超时分支      | 5    | 每缺失 1 个失败分支扣 2 分，扣完为止       |
| 每个判断节点有降级/重试策略         | 5    | 每缺失 1 个降级策略扣 2 分，扣完为止       |

#### E. [ASSUMPTION] 覆盖率（10 分）

| 检查项                                 | 分值 | 评分方法                               |
| -------------------------------------- | ---- | ------------------------------------------ |
| 推断内容是否被 `[ASSUMPTION]` 标记     | 5    | 已标记数 / 实际推断数 × 5                |
| 假设索引表是否汇总所有 `[ASSUMPTION]`  | 5    | 汇总完整得 5 分，每遗漏 1 条扣 2 分       |

#### F. 评审记录完整性（10 分）

| 检查项                               | 分值 | 评分方法                           |
| ------------------------------------ | ---- | -------------------------------------- |
| 第一性原理验证是否执行               | 3    | 输出固定格式得 3 分                     |
| 逻辑完整度审查是否执行               | 3    | 输出固定格式得 3 分                     |
| 边界与风险扫描是否执行               | 3    | 输出固定格式得 3 分                     |
| 评审记录固定格式是否完整             | 1    | 格式正确（含追溯通过率等）得 1 分        |

#### G. 产品思维深度（10 分）

| 检查项                     | 分值 | 评分标准                               |
| -------------------------- | ---- | -------------------------------------- |
| Why Now 是否清晰           | 2    | 明确阐述了问题紧急性和时间窗口得 2 分   |
| 差异化价值是否明确         | 2    | 清晰说明了与现有方案或竞品的区别得 2 分 |
| 用户分层是否合理           | 2    | 目标用户和次要用户有优先级区分得 2 分   |
| 商业价值是否阐述           | 2    | 说明了功能带来的商业收益或效率提升得 2 分 |
| 风险预判是否全面           | 2    | 识别了 ≥2 种技术风险并有缓解方案得 2 分 |
