# 术语表与 [ASSUMPTION] 标签规则

## 术语表（Glossary）规则

### 定义标准

跨章节出现 **≥2 次**的领域名词必须在第 11 章术语表中定义一次。

**领域名词示例**：VR 项目、自动填充、竞态、parse_status、pending_completion、降级策略

**非领域名词**（不需要定义）：用户、系统、前端、后端、数据库（通用技术术语）

### 定义格式

```markdown
- **术语** — 定义。与其他术语的关系（如适用）。基数关系（如适用）。
```

示例：
```markdown
- **VR 项目** — 用户在 App 端拍摄的 VR 看房项目，包含 GPS 坐标、项目原始名称、3D 模型等数据。与 project_id 一对一关联。
- **自动填充** — 系统根据 VR 项目的位置解析结果，自动填写搜索表单中的省市和小区名字段。
- **竞态** — 静默加载期间用户已开始手动输入的场景，此时自动填充结果不覆盖用户输入。
- **parse_status** — 位置解析状态枚举，取值：pending（未完成）/ success（全部成功）/ partial（部分成功）/ failed（全部失败）。
```

### 术语一致性

定义后，全文使用该术语必须**字面一致**：
- 禁止同义词替换（如定义了"VR 项目"，不能在某处写"VR 房源"）
- FR、US、埋点中使用术语时与术语表字面一致

**为什么**：下游读者（开发、测试、PM）需要精确理解每个术语的含义，同义词会导致歧义。

### 候选名词提取

生成 PRD 后，自动扫描文档：
1. 提取所有名词（排除通用词：用户、系统、功能、数据、接口等）
2. 统计出现次数（≥2 次）
3. 列为候选术语表，向用户展示
4. 用户确认哪些入表

## [ASSUMPTION] 标签规则

### 何时使用

LLM 在没有用户明确确认的情况下推断的内容，必须用 `[ASSUMPTION: 推断内容]` 内联标记。

**必须标记的情况**：
- 推断用户反馈/痛点（无具体数据支撑）
- 推断技术选型（如"使用 Redis 缓存"但用户未确认）
- 推断业务流程细节（如"假定上传后立即触发"）
- 推断量化目标（如"目标 >90%"但用户未给出具体值）

**不需要标记的情况**：
- 用户明确陈述的内容
- 从用户提供的文档/数据中直接提取的内容
- 行业标准做法（如"API 超时控制"）

### 格式

```
[ASSUMPTION: 推断内容]
```

示例：
```
触发时机：用户上传 VR 项目后 [ASSUMPTION: 假定上传后立即触发，而非用户手动确认后触发]
目标：自动填充成功率 >90% [ASSUMPTION: 量化目标基于行业基准，待用户确认]
```

### 假设索引（第 12 章）

PRD 生成/更新完成后，自动扫描全文 `[ASSUMPTION]` 标签，汇总到第 12 章：

```markdown
## 12. 假设索引

| # | 出处章节 | 假设内容 | 状态 |
|---|---------|---------|------|
| 1 | F-2.1 触发时机 | 假定上传后立即触发 | 待确认 |
| 2 | 第 2 章 成功指标 | 自动填充成功率 >90% 基于行业基准 | 待确认 |
```

**状态**：待确认 / 已确认 / 已否定

### 假设生命周期

1. **创建**：LLM 推断时插入 `[ASSUMPTION: xxx]`
2. **汇总**：生成/更新时自动汇总到第 12 章
3. **确认**：用户确认后，移除 `[ASSUMPTION]` 标签并从索引删除（状态改为"已确认"）
4. **否定**：用户否定后，移除标签并按用户意见修改内容（状态改为"已否定"）

**为什么**：假设是 PRD 中最大的风险源。显式标记和汇总确保用户审阅所有推断，避免 LLM 幻觉被当作需求。
